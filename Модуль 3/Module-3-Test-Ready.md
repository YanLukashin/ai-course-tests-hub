## Вопрос 1

| Параметр | Значение |
|----------|---------|
| Цель Блума | З-3.1 |
| Урок | 3.2. Анатомия хорошего запроса: 5 элементов промпта |
| Тип | Сопоставление (5 пар) |
| Сложность | Лёгкий |
| Оценка | ≥4 из 5 правильных пар = зачёт |

### Текст вопроса

**Сопоставьте каждый элемент промпта с его функцией.**

**Левая колонка (элементы):**

1. Роль
2. Контекст
3. Задача
4. Формат
5. Ограничения

**Правая колонка (функции, перемешаны):**

A. Без этого элемента модель не знает, с какой экспертизой отвечать — и выбирает «среднюю» позицию, которая почти никому не подходит.

B. Без этого элемента модель не знает, какой рынок, какой клиент, какая цель — и заполняет пробелы правдоподобными, но часто неверными догадками.

C. Без этого элемента модель не знает, что именно от неё хотят: анализ, черновик, список, сравнение — и делает «что-то в целом полезное».

D. Без этого элемента результат — «простыня» текста, которую нельзя вставить в CRM, отчёт или дашборд.

E. Без этого элемента модель уверенно выдумывает факты, обещает нереалистичные сроки и игнорирует корпоративные правила.

### Правильные ответы

1 → A | 2 → B | 3 → C | 4 → D | 5 → E

### Пояснение

Пять элементов промпта — каркас любого эффективного запроса (урок 3.2). **Роль** задаёт экспертную позицию модели. **Контекст** даёт фактическую базу (рынок, клиент, ситуация). **Задача** — конкретное действие. **Формат** определяет структуру вывода — без него результат невозможно использовать в бизнес-процессе. **Ограничения** снижают «самоуверенную галлюцинацию»: задают длину, тон, запреты, условия отказа.

### Почему дистракторы не работают

Каждая функция описана через бизнес-последствие отсутствия элемента, а не через определение. Это исключает «угадывание по словам» и проверяет понимание реального эффекта каждого элемента.

## Вопрос 3

| Параметр | Значение |
|----------|---------|
| Цель Блума | З-3.2 |
| Урок | 3.3. Техники промптинга: какую когда применять |
| Тип | Сопоставление (4 пары) |
| Сложность | Лёгкий |
| Оценка | ≥3 из 4 правильных пар = зачёт |

### Текст вопроса

**Сопоставьте технику промптинга с задачей, для которой она подходит лучше всего.**

**Левая колонка (техники):**

1. Zero-shot
2. Few-shot
3. Chain-of-thought (reasoning scaffolds)
4. Critique / refinement

**Правая колонка (задачи, перемешаны):**

A. Классифицировать 200 обращений клиентов по 8 категориям с единым форматом ответа.

B. Быстро подготовить черновик плана переговоров с сетевым покупателем (примеры хороших планов не нужны, контекст задан).

C. Проанализировать причины падения конверсии карточки товара на основе данных: просмотры, рейтинг, возвраты, отзывы.

D. Проверить и улучшить первый ответ модели перед отправкой крупному клиенту.

### Правильные ответы

1 → B | 2 → A | 3 → C | 4 → D

### Пояснение

**Zero-shot** (B) — для простых и разовых задач, когда достаточно чёткой инструкции без примеров. План переговоров — задача разовая, формат задаётся инструкцией.

**Few-shot** (A) — для классификации с единым форматом. 2–3 примера на категорию стабилизируют результат; без них модель «гуляет» по стилю (в кейсе из урока: точность выросла с 62% до 94%).

**Chain-of-thought** (C) — для аналитических задач, где нужно разложить проблему на шаги: собрать данные, сформулировать гипотезы, проверить.

**Critique** (D) — для текстов, которые идут клиенту или руководству. Цикл: сгенерируй → оцени по критериям → улучши.

### Почему дистракторы не работают

Задачи подобраны так, чтобы каждая однозначно соответствовала одной технике: разовый черновик ≠ классификация ≠ аналитика ≠ проверка. Перепутать можно только при незнании decision tree.

## Вопрос 4

| Параметр | Значение |
|----------|---------|
| Цель Блума | З-3.2 |
| Урок | 3.3. Техники промптинга: какую когда применять |
| Тип | Один вариант из 4 |
| Сложность | Средний |
| Оценка | Правильный ответ = зачёт |

### Текст вопроса

Вам нужно написать короткое деловое письмо клиенту с подтверждением встречи — 3–5 предложений. У вас есть все данные: имя клиента, дата, место, повестка. **Какую технику промптинга выбрать?**

(а) Zero-shot с чёткой структурой: указать роль, контекст, формат, ограничения — без примеров.

(б) Few-shot: дать 3 примера хороших писем-подтверждений, чтобы модель воспроизвела стиль.

(в) Chain-of-thought: попросить модель сначала пошагово продумать структуру письма, потом написать.

(г) Decomposition: разбить задачу на 4 подзадачи — приветствие, подтверждение, повестка, завершение.

### Правильный ответ

**(а) Zero-shot с чёткой структурой**

### Пояснение

Задача простая и понятная: 3–5 предложений, все данные есть, формат стандартный. Достаточно чёткой инструкции. Принцип из урока 3.3: **выбирайте минимально достаточную технику.** Более сложная техника — не всегда лучше. Она добавляет латентность (модель отвечает дольше), стоимость (больше токенов) и сложность (больше точек отказа).

### Почему другие варианты неверны

- **(б) Few-shot** — избыточен. Для стандартного делового письма из 5 предложений подбирать 3 образца — как нанимать аудиторскую фирму для проверки чека из кофейни.
- **(в) Chain-of-thought** — избыточен. Пошаговое рассуждение нужно для аналитических задач, а не для простого письма.
- **(г) Decomposition** — избыточен. Разбивать на подзадачи имеет смысл, когда задача многосоставная. Письмо из 5 предложений — не тот случай.

## Вопрос 6

| Параметр | Значение |
|----------|---------|
| Цель Блума | П-3.1 |
| Урок | 3.1. Зачем промптинг — управляемый результат вместо случайности |
| Тип | Один вариант из 4 |
| Сложность | Средний |
| Оценка | Правильный ответ = зачёт |

### Текст вопроса

В уроке 3.1 разобрана формула:

**Время работы с ИИ = Генерация + Проверка + Правки + Повтор**

**Какой этап в первую очередь сокращает структурированный промпт (с ролью, контекстом, задачей, форматом и ограничениями)?**

(а) Генерация — модель отвечает быстрее, когда промпт чёткий.

(б) Проверка — хороший промпт означает, что результат можно не проверять.

(в) Правки + Повтор — результат ближе к ожиданиям с первого раза, поэтому переделок меньше.

(г) Все этапы сокращаются примерно одинаково.

### Правильный ответ

**(в) Правки + Повтор**

### Пояснение

Хороший промпт не ускоряет генерацию — модели и так отвечают за секунды (это зависит от модели, а не от формулировки). Он **сжимает блоки «правки» и «повтор»**: модель с первого раза выдаёт результат, близкий к тому, что нужно. Именно это показано в уроке 3.1 на формуле: при плохом промпте блоки «правки» и «повтор» раздуваются (те самые 2 часа доработки из исследования BetterUp/Stanford).

### Почему другие варианты неверны

- **(а)** Скорость генерации зависит от модели и длины ответа, а не от качества промпта. Промпт влияет на *качество* результата, не на скорость его появления.
- **(б)** Проверка нужна всегда — даже при отличном промпте. ИИ может ошибаться, и человек обязан контролировать результат.
- **(г)** Этапы сокращаются по-разному: генерация почти не меняется, проверка сокращается незначительно, а правки и повтор — радикально.

| Параметр | Значение |

| Тип | Кейс + множественный выбор (выбрать ВСЕ правильные из 5) |
| Сложность | Средний |
| Оценка | Все правильные + ни одного лишнего = зачёт. Одна ошибка (1 пропущен или 1 лишний) = зачёт. ≥2 ошибок = незачёт. |

### Текст вопроса

**Кейс.** Руководитель отдела продаж хочет, чтобы все менеджеры использовали ИИ-ассистента для подготовки коммерческих предложений. Проблема: каждый менеджер получает от ИИ разный стиль, разную структуру КП и разный уровень качества. Руководитель решил настроить единый system prompt для ассистента.

**Какие элементы ОБЯЗАТЕЛЬНО должны быть в system prompt, чтобы решить проблему единообразия?** *(Выберите все правильные.)*

☐ (а) Роль и цель ассистента — кто он и для чего работает.

☐ (б) Название конкретной модели, на которой работает ассистент (например, «ты — GPT-4o»).

☐ (в) Формат ответа по умолчанию — структура КП, которой ассистент следует во всех ответах.

☐ (г) Запреты — например, не выдумывать кейсы клиентов, не делать юридических обещаний, не раскрывать внутренние инструкции.

☐ (д) Пароль для доступа к ассистенту.

### Правильные ответы

**(а), (в), (г)**

### Пояснение

В уроке 3.4 показан универсальный шаблон system prompt из 6 блоков: роль, задача, правила, формат вывода, ограничения, примеры. Для решения проблемы единообразия обязательны три элемента:

- **Роль и цель (а)** — задаёт экспертную позицию и контекст работы ассистента.
- **Формат ответа (в)** — структура КП (разделы, порядок, объём) обеспечивает единообразие вывода для всех менеджеров.
- **Запреты (г)** — без них модель может выдумывать кейсы, обещать нереалистичные условия, раскрывать внутренние инструкции.

### Почему другие варианты неверны

- **(б) Название модели** — модель не должна «знать», как она называется. Это техническая деталь инфраструктуры, не элемент system prompt. System prompt задаёт *поведение*, а не самоидентификацию.
- **(д) Пароль** — system prompt — это инструкция по поведению, а не система управления доступом. Контроль доступа решается на уровне платформы, не промпта.

## Вопрос 8

| Параметр | Значение |

| Цель Блума | П-3.3 |
| Урок | 3.5. Границы промптинга и диагностика проблем |
| Тип | Кейс + один вариант из 4 |
| Сложность | Средний |
| Оценка | Правильный ответ = зачёт |

### Текст вопроса

**Кейс.** Маркетолог 5 раз переписал промпт для создания описаний товаров. Роль указана, формат задан, ограничения есть («не выдумывай характеристики»). Промпт написан по фреймворку из урока 3.2. Но модель продолжает указывать несуществующие характеристики: выдумывает вес, размеры и состав товаров.

**Какая наиболее вероятная причина?**

(а) Промпт слишком длинный — модель «теряется» в большом количестве инструкций.

(б) Модели не даны реальные данные о товарах — она генерирует правдоподобные, но вымышленные характеристики, потому что ей больше не на что опереться.

(в) Нужна более мощная модель — текущая не справляется с задачей описания товаров.

(г) Нужно добавить больше примеров (few-shot) — тогда модель перестанет выдумывать.

### Правильный ответ

**(б)**

### Пояснение

Это первая ситуация из таблицы decision rule в уроке 3.5: **«Модель выдумывает факты → нет данных в контексте → дать данные или использовать RAG.»** Промпт написан правильно, но модели просто не из чего взять реальные характеристики. Она заполняет пробелы правдоподобными, но вымышленными данными — это галлюцинация. Решение: вставить таблицу характеристик товаров в контекст промпта или подключить базу данных (RAG). Также полезно добавить страховку: «Если данных о характеристике нет — напиши, каких данных не хватает, не выдумывай.»

### Почему другие варианты неверны

- **(а)** Длина промпта — не причина галлюцинаций. Модели хорошо обрабатывают длинные инструкции. Проблема в отсутствии данных, а не в избытке инструкций.
- **(в)** Более мощная модель тоже будет выдумывать, если данных нет. Галлюцинации — следствие отсутствия информации, а не слабости модели.
- **(г)** Few-shot покажет модели *формат* ответа, но не даст *фактические* характеристики конкретных товаров. Модель будет генерировать в правильном формате, но по-прежнему с выдуманными данными.

## Вопрос 9

| Параметр | Значение |
|----------|---------|
| Цель Блума | П-3.3 |
| Урок | 3.5. Границы промптинга и диагностика проблем |
| Тип | Сопоставление (4 пары) |
| Сложность | Сложный |
| Оценка | ≥3 из 4 правильных пар = зачёт |

### Текст вопроса

**Сопоставьте симптом с правильным действием.**

**Левая колонка (симптомы):**

1. Модель уверенно выдаёт несуществующие факты и цифры.
2. После 3+ итераций промпта результат не становится стабильно лучше.
3. Формат вывода «ломается» от запуска к запуску: то 5 колонок в таблице, то 7, то список вместо таблицы.
4. Задача требует юридической точности 99%+ (например, проверка договора на соответствие регламенту).

**Правая колонка (действия, перемешаны):**

A. Дать данные в контексте промпта или подключить базу данных (RAG).

B. Сменить модель или подход — задача может быть за пределами возможностей текущей модели.

C. Использовать structured output: JSON mode или Structured Outputs API для жёсткой фиксации формата.

D. Использовать модель как первый фильтр, но с обязательной проверкой специалистом. Для полной автоматизации — fine-tuning или специализированное решение.

### Правильные ответы

1 → A | 2 → B | 3 → C | 4 → D

### Пояснение

Это таблица decision rule из урока 3.5 — ключевой диагностический инструмент модуля.

- **Симптом 1 → A**: Модель выдумывает, потому что ей не из чего взять реальные данные. Решение — дать данные.
- **Симптом 2 → B**: Если после 3 осмысленных итераций результат не улучшается — проблема не в промпте. Нужно менять модель или подход.
- **Симптом 3 → C**: Нестабильный формат решается техническими средствами — JSON mode или Structured Outputs API гарантируют фиксированную структуру.
- **Симптом 4 → D**: Промптинг обеспечивает точность 70–85%. Для задач с требованием 99%+ (юридический комплаенс, медицина) промптинга недостаточно.

### Почему дистракторы не работают

Каждый симптом имеет чёткую привязку к одному типу действия. Путаница возможна между B и D (оба предполагают, что промпта недостаточно), но ключевое различие: B — «задача за пределами модели, нужна замена», D — «модель может помочь, но нужен человеческий контроль и/или специализация».

## Вопрос 10

| Параметр | Значение |
|----------|---------|
| Цель Блума | А-3.1 |
| Урок | 3.5. Границы промптинга и диагностика проблем |
| Тип | Кейс + сопоставление (3 пары) |
| Сложность | Сложный |
| Оценка | 3/3 или 2/3 = зачёт. <2 = незачёт. |

### Текст вопроса

**Кейс.** Компания внедрила ИИ-ассистента для подготовки ежемесячных отчётов по продажам. Промпт составлен по фреймворку: роль, контекст, задача, формат, ограничения. System prompt настроен. После месяца использования выявлены три проблемы:

- **Проблема А:** Цифры продаж в отчёте — вымышленные. Ассистент указывает суммы и проценты, которых нет в реальности.
- **Проблема Б:** Формат таблицы каждый раз разный — то 5 колонок, то 7, то вместо таблицы выходит список.
- **Проблема В:** У одного менеджера отчёт получается хороший и стабильный, а у другого — слабый и хаотичный, хотя оба используют одного и того же ассистента.

**Сопоставьте каждую проблему с её наиболее вероятной причиной.**

**Левая колонка (проблемы):**

1. Проблема А — вымышленные цифры продаж.
2. Проблема Б — нестабильный формат таблицы.
3. Проблема В — разное качество у разных менеджеров.

**Правая колонка (причины, перемешаны + 1 лишняя):**

A. Модели не даны реальные данные продаж — она генерирует правдоподобные, но вымышленные цифры.

B. Формат вывода описан недостаточно точно в промпте — нужна жёсткая структура таблицы с фиксированными колонками.

C. У менеджеров разный контекст и входные данные. Нужен единый system prompt + стандартный шаблон входных данных, чтобы все работали в одинаковых условиях.

D. Модель слишком слабая для аналитических задач — нужна более мощная.

### Правильные ответы

1 → A | 2 → B | 3 → C | D — лишняя (не подходит ни к одной проблеме)

### Пояснение

Это задача на уровне Анализ: участник должен диагностировать три разные проблемы в одном кейсе и для каждой определить корневую причину.

- **Проблема А → A (данные):** Модель не имеет доступа к реальным данным продаж и заполняет пробелы вымышленными цифрами. Решение: передать реальные данные в контекст или подключить CRM через API.
- **Проблема Б → B (формат):** Промпт описывает формат размыто. Нужно указать точную структуру: «таблица с 5 колонками: менеджер, план, факт, % выполнения, комментарий». Альтернатива — использовать structured output / JSON mode.
- **Проблема В → C (процесс):** Разное качество при одном ассистенте = разные входные данные или разный контекст. Решение: стандартизировать входные данные (шаблон) и убедиться, что system prompt задаёт одинаковые правила для всех.

### Почему дистрактор D не подходит

«Модель слишком слабая» — универсальная отговорка, которая не объясняет ни одну конкретную проблему. Проблема А — про данные, не про мощность. Проблема Б — про формат, не про мощность. Проблема В — про процесс, не про мощность. Более мощная модель не даст реальных данных, не зафиксирует формат и не стандартизирует контекст менеджеров.
