## Вопрос 1

| Параметр | Значение |
|----------|---------|
| Тип | Сопоставление (5 пар) |
| Цель Блума | З-5.1 |
| Урок | 5.2. Архитектура конвейера |
| Сложность | Лёгкий |

### Текст вопроса

Сопоставьте каждый элемент каркаса конвейера с его функцией.

### Элементы (левая колонка)

1. Trigger
2. Processing
3. Decision
4. Output
5. Контроль

### Описания функций (правая колонка)

A. Журнал событий, алерты при аномалиях, запасной сценарий при сбое — наблюдаемость всего конвейера.

B. Событие, расписание или условие, по которому запускается конвейер.

C. Точка принятия решения: пороги уверенности, эскалация на человека (HITL), ветвление по условиям.

D. Конкретный результат работы конвейера: запись в CRM, уведомление, обновление статуса — куда, в каком формате, какой статус.

E. Обработка данных: rule-блоки (фильтры, проверки) и AI-блоки (классификация, извлечение, генерация).

### Правильный ответ

| Элемент | Функция |
|---------|---------|
| 1 — Trigger | B |
| 2 — Processing | E |
| 3 — Decision | C |
| 4 — Output | D |
| 5 — Контроль | A |

### Пояснение

В уроке 5.2 каркас конвейера разобран как универсальная схема любой автоматизации: trigger запускает процесс, processing обрабатывает (rule + AI), decision определяет дальнейший путь через пороги и HITL, output фиксирует результат, а контроль охватывает всё снизу — логи, алерты, fallback.

### Почему другие комбинации неверны

- Trigger легко спутать с Decision, но trigger — это только «запуск», а не «принятие решения по результатам».
- Processing и Decision путают чаще всего: обработка — это *работа с данными*, а decision — это *выбор действия на основе результата*.
- Контроль — единственный элемент, который идёт «снизу», охватывая весь конвейер (логи, алерты, fallback).

**Зачёт:** ≥ 4 из 5 пар верны.

## Вопрос 2

| Параметр | Значение |
|----------|---------|
| Тип | Один вариант из 5 |
| Цель Блума | З-5.1 |
| Урок | 5.2. Архитектура конвейера |
| Сложность | Лёгкий |

### Текст вопроса

Ваша автоматизация классифицирует входящие обращения клиентов по теме (техническая проблема, возврат, доставка, жалоба). В каком элементе каркаса конвейера размещается AI-блок, выполняющий эту классификацию?

### Варианты ответов

A. Trigger
B. Processing
C. Decision
D. Output
E. Контроль

### Правильный ответ

**B. Processing**

### Пояснение

В уроке 5.2 прямо сказано: «ИИ живёт внутри processing. Он не заменяет всю схему. Это один или два станка на конвейере, не весь завод.» AI-блок — это часть обработки, а не запуска, решения или вывода результата.

### Почему другие варианты неверны

- **A (Trigger)** — trigger только запускает конвейер (событие, расписание, условие), но не обрабатывает данные.
- **C (Decision)** — decision определяет действие по результатам обработки (пороги, HITL), но сам не классифицирует.
- **D (Output)** — output фиксирует результат (запись в CRM, уведомление), но не обрабатывает входные данные.
- **E (Контроль)** — контроль обеспечивает наблюдаемость (логи, алерты, fallback), но не выполняет бизнес-логику.

3. Клиент загружает скан счёта-фактуры. IDP-система извлекает поля (номер, дата, сумма, ИНН), rule-блок валидирует по справочнику контрагентов, при совпадении — автоматическая запись в ERP.

4. Поступило обращение в helpdesk. LLM определяет тему, rule-блок проверяет SLA клиента, при confidence ≥ 85% — тикет автоматически назначается нужному специалисту.

A. Классификация и маршрутизация входящих
B. Обработка документов
C. Генерация отчётов и дайджестов
D. Обогащение данных
E. Маршрутизация внутренних задач *(дистрактор)*

| Сценарий | Паттерн |

| 1 | C — Генерация отчётов и дайджестов |
| 2 | D — Обогащение данных |
| 3 | B — Обработка документов |
| 4 | A — Классификация и маршрутизация входящих |

### Пояснение

Все пять паттернов разобраны в уроке 5.2 через каркас конвейера. Каждый описан с конкретным trigger, processing, decision, output. Маршрутизация внутренних задач — реальный паттерн (trigger: новая задача, AI определяет тип, rule-блок сверяет с матрицей ответственности). Но ни один из четырёх описанных сценариев ему не соответствует.

## Вопрос 4

| Параметр | Значение |

| Тип | Сопоставление (4 пары) |
| Цель Блума | З-5.2 |
| Урок | 5.4. Операционные контроли |
| Сложность | Лёгкий |

### Текст вопроса

Сопоставьте каждый операционный контроль с его назначением.

### Контроли (левая колонка)

1. HITL (human-in-the-loop)
2. Логирование
3. Fallback
4. Sandbox

### Назначения (правая колонка)

A. Запасной сценарий: определяет, что делает система, когда AI-блок ошибается или недоступен.

B. Запись всего, что произошло на каждом шаге: вход, выход AI, решение, trace-id — возможность расследовать инцидент.

C. Человек проверяет решение AI до того, как оно будет применено — перехват ошибок на критичных шагах.

D. Изолированная среда с минимальными правами: AI-блок не может выполнить действия за пределами разрешённого набора.

### Правильный ответ

| Контроль | Назначение |
|----------|-----------|
| 1 — HITL | C |
| 2 — Логирование | B |
| 3 — Fallback | A |
| 4 — Sandbox | D |

### Пояснение

В уроке 5.4 каждый из четырёх контролей разобран подробно. HITL — «человек в контуре, проверяет решение ИИ до того, как оно применится». Логи — «бортовой самописец: без него расследование невозможно». Fallback — «запасной генератор в больнице: вопрос не "если сломается", а "когда"». Sandbox — «ключи от кабинетов: уборщик не получает ключ от сейфа».

### Почему другие комбинации неверны

- HITL ≠ логирование: HITL *предотвращает* ошибку, логи *фиксируют*, что произошло.
- Fallback ≠ sandbox: fallback — это *план Б при сбое*, sandbox — это *ограничение прав доступа*.

**Зачёт:** ≥ 3 из 4 пар верны.

## Вопрос 5

| Параметр | Значение |
|----------|---------|
| Тип | Один вариант из 4 |
| Цель Блума | З-5.2 |
| Урок | 5.3. Выбор платформы |
| Сложность | Средний |

### Текст вопроса

Страховая компания выбирает no-code платформу для автоматизации обработки заявлений. В заявлениях — персональные данные клиентов: ФИО, паспортные данные, медицинская информация.

Какой из критериев выбора платформы является **hard constraint** (абсолютным стопфактором) в этой ситуации?

### Варианты ответов

A. Интеграции — наличие коннекторов к CRM и системе документооборота компании

B. Скорость запуска — время до первого рабочего сценария

C. Контроль данных и регион хранения — где физически хранятся данные, возможность self-hosted

D. Стоимость сценария — цена за один прогон конвейера с учётом AI-вызовов

### Правильный ответ

**C. Контроль данных и регион хранения**

### Пояснение

В уроке 5.3 прямо сказано: «Если ваш сценарий обрабатывает персональные данные клиентов — контроль данных становится абсолютным стопфактором. Платформа, которая не отвечает на вопрос "где хранятся данные" — не подходит для этого сценария, даже если по остальным четырём критериям она идеальна.» NIST AI RMF подтверждает: контроль над данными — базовый элемент управления рисками.

### Почему другие варианты неверны

- **A (Интеграции)** — важный критерий, но это optimizer: недостающий коннектор можно обойти через промежуточный вебхук. Отсутствие контроля над ПДн обойти нельзя.
- **B (Скорость запуска)** — в сценарии 5.3 прямо назван optimizer, а не hard constraint: «скорость запуска — это optimizer, а не абсолютный стопфактор».
- **D (Стоимость сценария)** — важна для ROI, но не является стопфактором при работе с чувствительными данными. Дешёвая платформа без контроля данных — хуже дорогой с контролем.

## Вопрос 6

| Параметр | Значение |
|----------|---------|
| Тип | Множественный выбор (несколько верных) |
| Цель Блума | П-5.1 |
| Урок | 5.2. Архитектура конвейера |
| Сложность | Средний |

### Текст вопроса

Компания автоматизирует обработку входящих заявок от клиентов. Ниже перечислены 5 шагов конвейера. **Отметьте все шаги, которые являются rule-блоками** (детерминированными, с предсказуемым результатом).

### Варианты (отметить все подходящие)

A. Проверить: сумма заявки > 100 000 руб. → отправить на согласование

B. Классифицировать текст обращения по теме (техническая проблема, возврат, доставка, жалоба)

C. Отформатировать JSON-ответ по фиксированному шаблону для записи в CRM

D. Извлечь тему и ключевые слова из текста письма клиента

E. Записать результат обработки в CRM, если статус = «одобрено»

### Правильный ответ

**A, C, E**

### Пояснение

В уроке 5.2 разделение определено так: rule-блок — «калькулятор: 2 + 2 всегда 4» (проверки, фильтры, валидации, форматирование по шаблону, запись по условию). AI-блок — «опытный менеджер, который делает вывод из контекста» (классификация, извлечение, саммари, генерация).

- **A** — сравнение суммы с порогом: детерминированная проверка → rule.
- **C** — форматирование по фиксированному шаблону: всегда один результат → rule.
- **E** — запись по условию (статус = «одобрено»): детерминированная логика → rule.

### Почему другие варианты не rule

- **B** — классификация текста по теме требует смыслового анализа, формат обращений каждый раз разный → AI-блок.
- **D** — извлечение темы и ключевых слов из свободного текста — «найти в тексте смысл» → AI-блок.

**Зачёт:** отмечены все три верных (A, C, E) и ни одного лишнего.

## Вопрос 7

| Параметр | Значение |
|----------|---------|
| Тип | Кейс с вариантами ответа |
| Цель Блума | П-5.2 |
| Урок | 5.1. Когда автоматизировать, а когда нет |
| Сложность | Средний |

### Текст вопроса

Прочитайте описание процесса и решите: стоит ли его автоматизировать прямо сейчас?

**Кейс.** Компания хочет автоматизировать согласование командировок.

Факты:
- Частота: 8 заявок в месяц (примерно 2 в неделю).
- SOP: не задокументирован. Каждый менеджер оформляет по-своему — кто-то пишет в почту, кто-то в мессенджер, кто-то заполняет таблицу.
- Формат заявок: произвольный.
- Владелец процесса: не назначен.
- Метрика «после»: не определена.

### Варианты ответов

A. Автоматизировать: частота достаточная (8 в месяц — это регулярный процесс), остальные проблемы решатся по ходу.

B. Автоматизировать, но с HITL на каждом шаге — это компенсирует отсутствие SOP и владельца.

C. Не автоматизировать сейчас: процесс не проходит скрининг — нарушены минимум два критерия (нет SOP, нет владельца). Сначала стандартизировать формат заявок, назначить владельца, описать процедуру.

D. Нужно собрать больше данных, прежде чем принимать решение — информации недостаточно.

### Правильный ответ

**C**

### Пояснение

В уроке 5.1 этот кейс разобран как контрпример скрининга. По 5-вопросному тесту: частота — пограничная (8/мес., ✗ или под вопросом); SOP не задокументирован (✗); владелец не назначен (✗). Правило из урока: «Два или больше "нет" — остановитесь. Не тратьте деньги. Сначала опишите SOP. Назначьте владельца. Стандартизируйте входы.»

### Почему другие варианты неверны

- **A** — 8 заявок в месяц — это пограничная частота, а главное — «остальные проблемы решатся по ходу» противоречит принципу из урока: «сначала процесс, потом автоматизация». ИИ масштабирует хаос, не устраняет.
- **B** — HITL на каждом шаге = ручной процесс с дополнительными шагами. В уроке 5.4 прямо сказано: «HITL на каждый шаг — это не контроль, это ручной процесс с дополнительными шагами. Зачем тогда автоматизация?»
- **D** — информации вполне достаточно: мы знаем частоту, отсутствие SOP, отсутствие владельца. Два критерия скрининга не пройдены — вывод однозначен.

## Вопрос 8

| Параметр | Значение |
|----------|---------|
| Тип | Сопоставление (4 пары) |
| Цель Блума | П-5.3 |
| Урок | 5.4. Операционные контроли |
| Сложность | Средний |

### Текст вопроса

Для каждого отсутствующего контроля укажите конкретное операционное последствие.

### Отсутствующий контроль (левая колонка)

1. Нет логирования
2. Нет fallback
3. Нет HITL
4. Нет sandbox

### Последствия (правая колонка)

A. При ошибке AI система зависает или выдаёт пустой результат клиенту — нет запасного сценария.

B. Ошибки AI применяются автоматически, включая дорогостоящие — человек не может перехватить решение до его исполнения.

C. Невозможно расследовать инцидент: непонятно, что AI получил на вход, что ответил и почему решение было применено.

D. AI-инструмент может получить избыточные права и выполнить непреднамеренное действие — удалить данные, отправить лишнее, обновить не ту запись.

### Правильный ответ

| Отсутствующий контроль | Последствие |
|-----------------------|-------------|
| 1 — Нет логирования | C |
| 2 — Нет fallback | A |
| 3 — Нет HITL | B |
| 4 — Нет sandbox | D |

### Пояснение

В уроке 5.4 последствия отсутствия каждого контроля разобраны на кейсе рассылки. Без логов — «расследование невозможно, три дня инцидента без обнаружения». Без fallback — «при ошибке AI письмо уходит с пустым полем скидки». Без HITL — «все письма с любой суммой уходят автоматически, ошибки никто не перехватывает». Без sandbox — OWASP LLM01 (prompt injection) и LLM08 (excessive agency): модель может выполнить действие, которого никто не просил.

### Почему другие комбинации неверны

- «Нет логирования → ошибки применяются автоматически» — неверно: логи *фиксируют*, но не *предотвращают*. Предотвращение — это HITL.
- «Нет fallback → избыточные права» — неверно: fallback про *план Б при сбое*, а не про безопасность доступа.

**Зачёт:** ≥ 3 из 4 пар верны.

| Сложность | Сложный |

### Текст вопроса

Прочитайте описание автоматизации. Отметьте **все** архитектурные ошибки, которые присутствуют в схеме.

**Описание автоматизации: «Персонализированная рассылка предложений»**

Trigger: менеджер загружает список клиентов в Google Sheets.

Processing: один AI-блок получает список и для каждого клиента генерирует персонализированный текст письма, включая имя, описание товара и сумму скидки. Тот же AI-блок проверяет, что сумма скидки соответствует прайсу компании.

Decision: нет. Все сгенерированные письма уходят автоматически.

Output: письма отправляются клиентам.

Контроль: в описании не упомянуты логи, алерты, fallback.

### Варианты (отметить все подходящие)

A. Смешение rule-блока и AI-блока: проверка суммы по прайсу — это детерминированная операция (rule), но она поручена AI-блоку вместе с генерацией текста.

B. Нет fallback: если AI вернёт ошибку или пустой результат — письмо уйдёт с некорректным содержимым, запасного сценария нет.

C. Нет HITL: все письма отправляются автоматически, без проверки человеком — включая письма с крупными суммами скидок.

D. Нет логирования: невозможно восстановить, какие письма ушли, с каким содержимым и почему.

E. Нет confidence threshold: AI-блок не оценивает уверенность в результате. *(Дистрактор)*

F. Неверный trigger: загрузка списка в Google Sheets — некорректный способ запуска. *(Дистрактор)*

### Правильный ответ

**A, B, C, D**

### Пояснение

Все четыре ошибки прямо разобраны в уроке 5.4 на этом же кейсе (составной портрет ошибок рассылки):

1. **Смешение rule/AI (A):** проверка суммы по прайсу — детерминированная операция, но поручена AI-блоку. В уроке 5.2: «Вот формула: rule-блок — предсказуемость. AI-блок — гибкость. Смешение — непредсказуемая гибкость, то есть хаос.»

2. **Нет fallback (B):** в уроке 5.4 разобрано: «Модель начинает галлюцинировать в числах. Вместо скидки 10% пишет 80%.» Нет запасного сценария — письмо уходит с ошибкой.

3. **Нет HITL (C):** в уроке 5.4: «Если бы стоял value-based гейт — все письма с суммой скидки > 50 000 рублей на проверку менеджеру — проблему перехватили бы до отправки.»

4. **Нет логов (D):** «Инцидент обнаружили через три дня — по жалобам клиентов. Сколько писем ушло с ошибкой? Неизвестно. Откатить? Невозможно.»

### Почему E и F — не ошибки

- **E (нет confidence threshold)** — confidence threshold важен, но в описании он не является самостоятельной ошибкой: в данной схеме проблема шире — нет вообще никакого механизма проверки качества (нет HITL, нет decision). Confidence threshold — частный случай, а не корневая ошибка.
- **F (неверный trigger)** — загрузка списка в Google Sheets — вполне корректный trigger (событие: файл загружен). Проблема не в том, как конвейер запускается, а в том, что происходит после.

**Зачёт:** отмечены все четыре верных (A, B, C, D) и ни одного лишнего. Допустим частичный балл: −1 за каждую ошибку (пропущенный верный или выбранный лишний).

## Вопрос 10

| Параметр | Значение |
|----------|---------|
| Тип | Кейс с вариантами ответа |
| Цель Блума | А-5.1 |
| Уроки | 5.2, 5.4 |
| Сложность | Сложный |

### Текст вопроса

Используя ту же автоматизацию рассылки из вопроса 9:

Одна из ошибок — **нет HITL: все письма с любой суммой скидки отправляются автоматически без проверки человеком**. Менеджер узнаёт об ошибках только по жалобам клиентов.

Какое исправление **архитектурно корректно** решает эту проблему?

### Варианты ответов

A. Добавить HITL на все письма без исключения: каждое письмо перед отправкой проверяет менеджер.

B. Добавить value-based HITL-гейт: письма с суммой скидки выше установленного порога (например, > 50 000 руб.) уходят на подтверждение менеджеру с SLA 2 часа; остальные — автоматически.

C. Добавить логирование всех отправленных писем: после отправки можно будет найти ошибочные.

D. Добавить fallback: при ошибке AI не отправлять письмо — поставить в очередь.

### Правильный ответ

**B**

### Пояснение

В уроке 5.4 разобраны три типа HITL-гейтов: confidence-based, value-based, exception-based. Для рассылки с суммами скидок подходит именно value-based: «Тут неважно, уверена модель или нет. Важна цена ошибки. Если автоматизация готовит действие, которое стоит дорого — человек подтверждает, прежде чем это уйдёт.»

Value-based HITL с конкретным порогом, SLA и назначенным рецензентом — это архитектурно грамотное решение: рутинные письма уходят автоматически (автоматизация работает), а рискованные — проверяются (ошибки перехватываются).

### Почему другие варианты неверны

- **A** — HITL на все письма без исключения уничтожает смысл автоматизации. В уроке 5.4: «HITL на каждый шаг — это не контроль, это ручной процесс с дополнительными шагами.»
- **C** — логирование позволит *расследовать* инцидент, но не *предотвратить* его. Ошибочные письма уже ушли клиентам. Логи не заменяют HITL.
- **D** — fallback защищает от *сбоя AI* (ошибка, пустой ответ), но проблема здесь в том, что AI *работает* — просто генерирует неверные числа. Fallback не срабатывает, когда AI возвращает ответ (пусть и неверный).
