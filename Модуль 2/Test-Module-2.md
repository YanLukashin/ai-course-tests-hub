## Вопрос 1

| Параметр | Значение |
|----------|---------|
| **Тип** | Сопоставление (5 пар) |
| **Цель Блума** | З-2.1 |
| **Урок** | 2.2 — Классы моделей: что реально покупает бизнес |
| **Сложность** | Лёгкий |
| **Критерий зачёта** | Правильно сопоставлены минимум 4 из 5 пар |

### Задание

Сопоставьте класс решения (левый столбец) с бизнес-ситуацией, для которой он наиболее оправдан (правый столбец).

**Левый столбец — классы решений:**

- А. Frontier general-purpose
- Б. Workhorse / mini / fast
- В. Open-weight / self-hosted
- Г. Специализированные / domain-tuned
- Д. Privacy-first / локальные платформы

**Правый столбец — бизнес-ситуации:**

1. У компании строгий комплаенс: данные клиентов не могут покидать страну, нужна модель с прозрачной политикой хранения данных на территории РФ
2. Нужно ежедневно суммаризировать 200 звонков: сначала распознать речь, затем сгенерировать краткое резюме каждого разговора
3. Компания разрабатывает агентный сценарий, где модель должна планировать цепочку из 10–15 шагов, анализировать сложные документы и принимать решения
4. Нужно классифицировать 1 000 обращений клиентов в день по типу — задача простая и однотипная, критична стоимость
5. Компания хочет полный контроль над моделью: запускать на своих серверах, адаптировать под свои процессы, есть команда для поддержки инфраструктуры

### Правильные пары

- А → 3
- Б → 4
- В → 5
- Г → 2
- Д → 1

### Пояснение

Каждый класс закрывает свой профиль задач. Frontier — для задач, где нужны сложные рассуждения и планирование (агентные сценарии). Workhorse/mini — для массовых простых задач, где критична стоимость. Open-weight — когда нужен полный контроль и есть команда поддержки. Специализированные — когда задача решается комбинацией узких компонентов (ASR + LLM). Privacy-first/локальные — когда жёсткий комплаенс по данным в РФ.

### Почему другие пары не подходят

- Frontier для массовой классификации (ситуация 4) — переплата: «как заказывать доставку пиццы на бронированном лимузине» (урок 2.1).
- Workhorse для агентных цепочек (ситуация 3) — не хватит «интеллекта» для многошагового планирования.
- Privacy-first для полного контроля (ситуация 5) — privacy-first платформы решают вопрос данных, но не дают полный контроль над инфраструктурой, как self-hosted.
- Open-weight для комплаенса по данным (ситуация 1) — open-weight требует своей команды и инфраструктуры; если задача только в хранении данных в РФ — проще использовать готовую локальную платформу.
- Специализированные для комплаенса (ситуация 1) — специализированные модели решают узкие задачи (ASR, OCR), а не вопрос хранения данных.

| **Сложность** | Лёгкий |
| **Критерий зачёта** | 1 правильный ответ |

### Вопрос

Компания работает с персональными данными клиентов. По требованиям ФЗ-152 данные не могут покидать территорию РФ. Какой класс решений следует рассматривать в первую очередь?

### Варианты ответов

- **A.** Frontier-модель глобального вендора через API — она самая «умная» и точно справится лучше
- **B.** Workhorse-модель западного провайдера — она дешёвая и быстрая
- **C.** Privacy-first / локальная платформа в РФ-контуре или open-weight модель на своих серверах
- **D.** Специализированная ASR-модель для распознавания речи

### Правильный ответ

**C**

### Пояснение

Когда данные не могут покидать РФ — это hard constraint по приватности и доступности. Он отсекает все варианты, где данные уходят за рубеж. Остаются два пути: локальная платформа (GigaChat, YandexGPT — данные в РФ) или open-weight модель, развёрнутая на своих серверах в российском контуре. Именно это разобрано в уроке 2.2, блок «Privacy-first».

### Почему другие неверны

- **A** — frontier-модели глобальных вендоров обрабатывают данные за рубежом, а ряд провайдеров не включает Россию в список поддерживаемых стран. Это нарушает hard constraint.
- **B** — workhorse западного провайдера тоже отправляет данные на зарубежные серверы. Дешевизна не снимает проблему комплаенса.
- **D** — ASR-модель распознаёт речь, а не решает задачу обработки персональных данных в целом. Это другой класс и другая задача.

## Вопрос 3

| Параметр | Значение |
|----------|---------|
| **Тип** | Сопоставление (8 критериев → 2 категории) |
| **Цель Блума** | З-2.2 |
| **Урок** | 2.1, 2.3 |
| **Сложность** | Средний |
| **Критерий зачёта** | Правильно распределены минимум 6 из 8 |

### Задание

Распределите критерии выбора модели на две категории: **Hard Constraints** (отсекают — если не выполнено, модель не подходит) и **Optimizers** (помогают выбрать лучшее из подходящего).

### Критерии для распределения

1. Сервис официально не поддерживает ваш регион — невозможно заключить договор
2. Модель отвечает на 2 секунды быстрее конкурента на аналогичных запросах
3. Данные клиентов не могут покидать территорию РФ по требованиям регуляторов
4. Стоимость токена на 20% ниже, чем у альтернативы
5. Для задачи нужен ввод изображений, а модель работает только с текстом
6. Удобнее SDK и документация для разработчиков
7. Требуется API с ответом в формате JSON — без этого автоматизация невозможна
8. Качество ответа чуть выше по результатам слепого теста

### Правильное распределение

**Hard Constraints:** 1, 3, 5, 7

**Optimizers:** 2, 4, 6, 8

### Пояснение

Hard constraints — бинарные: подходит или нет. Если сервис не работает в вашем регионе (1), данные нельзя отправлять за рубеж (3), нужна мультимодальность, а её нет (5), или автоматизация требует JSON, а модель его не выдаёт (7) — это стоп-факторы. Никакое «компенсирующее» преимущество их не перевешивает.

Optimizers — это «лучше/хуже», а не «можно/нельзя». Скорость на 2 секунды (2), цена на 20% (4), удобство SDK (6), небольшое преимущество по качеству (8) — всё это помогает выбрать лучшего из допустимых кандидатов, но не отсекает их.

### Почему другие распределения неверны

- «Скорость на 2 секунды» (2) — это не SLA-проблема, а оптимизация. Если бы задача требовала ответа за 1 секунду, а модель отвечала 15 — это был бы hard constraint. Но «чуть быстрее» — optimizer.
- «Стоимость на 20% ниже» (4) — экономия приятна, но не отсекает вариант. Если бы бюджет был жёстко ограничен, а модель в него не вписывалась — тогда hard constraint. Здесь речь о сравнительном преимуществе.
- «JSON через API» (7) — для кого-то это может показаться «технической фишкой», но если downstream-система ждёт структуру и без неё автоматизация невозможна — это стоп-фактор.

## Вопрос 4

| Параметр | Значение |
|----------|---------|
| **Тип** | Один вариант (1 из 4) |
| **Цель Блума** | З-2.2 |
| **Урок** | 2.3 — Критерии выбора и scorecard |
| **Сложность** | Средний |
| **Критерий зачёта** | 1 правильный ответ |

### Вопрос

Почему способность модели выдавать structured output (ответ в формате JSON по заданной схеме) считается бизнес-критерием, а не просто технической особенностью?

### Варианты ответов

- **A.** Потому что JSON — это современный стандарт, и его поддержка говорит о продвинутости модели
- **B.** Потому что structured output ускоряет генерацию текста и снижает время ответа модели
- **C.** Если downstream-система (CRM, ERP, таблица) ожидает данные в определённой структуре, а модель выдаёт невалидный формат — растут ретраи, ручная правка и стоимость интеграции
- **D.** Потому что бенчмарки учитывают способность модели следовать формату, и это влияет на позицию в рейтингах

### Правильный ответ

**C**

### Пояснение

В уроке 2.3 прямо говорится: «Structured output и предсказуемый формат — это не технические фишки, а бизнес-критерии. Если downstream-система ждёт JSON или табличную структуру — выбирайте модели, где это работает надёжно». Когда формат «плывёт», каждый невалидный ответ — это либо повторный запрос (ретрай = деньги), либо ручная правка (время сотрудника = деньги), либо сбой в автоматизации. Это прямой бизнес-убыток.

### Почему другие неверны

- **A** — «современность стандарта» не делает его бизнес-критерием. Бизнес-критерий — это влияние на стоимость и надёжность процесса.
- **B** — structured output не ускоряет генерацию. Время ответа — отдельный критерий (скорость/латентность).
- **D** — позиция в рейтингах не имеет прямого отношения к бизнес-ценности формата. Модуль учит, что бенчмарки — не замена собственному тесту.

## Вопрос 5

| Параметр | Значение |
|----------|---------|
| **Тип** | Кейс с вариантами ответа (1 из 4) |
| **Цель Блума** | П-2.1 |
| **Урок** | 2.1 — Почему нет лучшей модели вообще |
| **Сложность** | Средний |
| **Критерий зачёта** | 1 правильный ответ |

### Кейс

Руководитель компании говорит команде: «Давайте не будем усложнять — возьмём самую мощную модель на рынке. Она по всем рейтингам первая, значит, точно справится лучше всех с любой нашей задачей».

### Вопрос

Какой аргумент **лучше всего** объясняет, почему этот подход ошибочен?

### Варианты ответов

- **A.** Самая мощная модель слишком дорогая — бюджет не выдержит
- **B.** Оптимальный выбор зависит от задачи: стоимость сценария, доступность в вашем регионе, политика данных и качество на русском языке могут сделать менее мощную модель лучшим решением
- **C.** Рейтинги обновляются каждую неделю, поэтому сегодняшний лидер завтра будет вторым
- **D.** Мощные модели работают медленнее, а скорость важнее качества

### Правильный ответ

**B**

### Пояснение

Центральная мысль урока 2.1: «лучшая модель вообще» не существует. Выбор модели — это управленческий компромисс по нескольким осям. «Самая мощная» модель может быть недоступна в вашем регионе (hard constraint), отправлять данные за рубеж (hard constraint), стоить в разы дороже при простой задаче (неоптимальная экономика), или выдавать ошибки на русском (не протестирована на вашем языке). Менее мощная модель, которая проходит все hard constraints и решает задачу на «четвёрку», — рациональнее.

### Почему другие неверны

- **A** — «слишком дорогая» — верно, но это лишь одна ось из многих. Ответ неполный: даже если бюджет безграничен, остаются проблемы приватности, языка, доступности.
- **C** — «рейтинги меняются» — верно, но это аргумент про волатильность, а не про ошибочность подхода «берём лучшую». Даже если лидер стабилен — подход всё равно ошибочен.
- **D** — «медленнее» — не всегда так, и скорость не всегда важнее качества. Это частный случай, а не системный аргумент.

## Вопрос 6

| Параметр | Значение |
|----------|---------|
| **Тип** | Множественный выбор (нужно выбрать все правильные) |
| **Цель Блума** | П-2.2 |
| **Урок** | 2.4 — Экономика: стоимость сценария и когда платить за frontier |
| **Сложность** | Средний |
| **Критерий зачёта** | Минимум 2 из 3 правильных без лишних = зачёт |

### Вопрос

Что входит в «стоимость сценария» (cost per outcome) **помимо** цены за токены?

Выберите **все** подходящие варианты.

### Варианты ответов

- **A.** Стоимость ретраев — повторных запросов, когда модель ответила некорректно
- **B.** Стоимость обучения (дообучения) модели на ваших данных
- **C.** Стоимость ручной правки и проверки ответов сотрудниками
- **D.** Стоимость покупки GPU-серверов для размещения модели
- **E.** Стоимость ошибок — потерянный клиент, претензия, испорченный документ
- **F.** Стоимость маркетинга вашего ИИ-проекта внутри компании

### Правильные ответы

**A, C, E**

### Пояснение

В уроке 2.4 разобрана формула cost per outcome. Помимо стоимости токенов, в неё входят: ретраи (каждый повторный запрос = повторный расход токенов + время), ручная правка (время сотрудника = зарплата), и стоимость ошибок (потерянный клиент или претензия могут перекрыть экономию за полгода). В сценарии прямо сказано: «Стоимость билета — не стоимость поездки. Стоимость токена — не стоимость результата».

### Почему другие неверны

- **B** — дообучение модели — это отдельный проект, не часть стоимости использования через API. Для большинства бизнес-сценариев модуля (API-based) это нерелевантно.
- **D** — покупка GPU относится к self-hosted сценарию, а не к облачному API. Формула cost per outcome из урока 2.4 описывает стоимость для API-сценариев.
- **F** — маркетинг ИИ-проекта — организационный расход, не связанный со стоимостью конкретного сценария использования модели.

## Вопрос 7

| Параметр | Значение |
|----------|---------|
| **Тип** | Кейс с вариантами ответа (1 из 4) |
| **Цель Блума** | П-2.2 |
| **Урок** | 2.4 — Экономика: стоимость сценария и когда платить за frontier |
| **Сложность** | Средний |
| **Критерий зачёта** | 1 правильный ответ |

### Кейс

Компания тратит значительные суммы на frontier-модель для массовой классификации обращений клиентов. Объём — 1 000 обращений в день. Задача простая и однотипная: определить тему и срочность. Результат не нужен мгновенно — операторы разбирают обращения утром за предыдущие сутки.

### Вопрос

Какая комбинация мер **наиболее рациональна** для снижения стоимости?

### Варианты ответов

- **A.** Включить кэширование контекста — это даст экономию до 90%
- **B.** Перейти на дешёвую модель (workhorse/mini) и использовать batch-обработку (пакетную, с отложенным результатом)
- **C.** Настроить маршрутизацию: простые обращения — на дешёвую модель, сложные — на frontier
- **D.** Увеличить контекстное окно, чтобы обрабатывать больше обращений за один запрос

### Правильный ответ

**B**

### Пояснение

В кейсе два ключевых условия: (1) задача простая и однотипная — frontier избыточен, workhorse/mini справится, и (2) результат не нужен мгновенно — идеальные условия для batch-обработки со скидкой ~50%. Комбинация «дешёвая модель + batch» даёт максимальный эффект. Это прямо разобрано в уроке 2.4: рычаг 2 (batch — «не срочно? экономия порядка 50%») и рычаг 3 / блок 5 (когда дешёвая модель рациональнее — «объём + низкая цена ошибки»).

### Почему другие неверны

- **A** — кэширование экономит на повторяющемся контексте (system prompt), но не решает главную проблему: компания платит за frontier, когда задача простая. Экономия на кэше не компенсирует переплату за класс модели.
- **C** — маршрутизация имеет смысл, когда часть запросов сложные, а часть простые. Но в кейсе все 1 000 обращений — простые и однотипные. Держать frontier для «сложных» не нужно — их нет.
- **D** — увеличение контекстного окна увеличивает стоимость (context window tax из урока 2.4), а не снижает её.

## Вопрос 8

| Параметр | Значение |

| **Урок** | 2.5 — Слепой тест моделей: как выбрать без демо-иллюзий |
| **Сложность** | Лёгкий |
| **Критерий зачёта** | 4 из 5 шагов на правильных позициях |

### Задание

Расставьте шаги алгоритма слепого теста моделей в правильном порядке.

### Шаги (в перемешанном порядке)

- ☐ Переименовать ответы моделей в «Вариант A / B / C» и оценить вслепую по четырём метрикам
- ☐ Составить shortlist из 2–4 моделей-кандидатов, прошедших hard constraints
- ☐ Подготовить единый тест-промпт (одинаковые инструкции для всех моделей)
- ☐ Выбрать основную модель и fallback (запасной вариант)
- ☐ Собрать тестсет из 3–5 реальных кейсов (70% типичных + 30% краевых)

### Правильный порядок

1. Составить shortlist из 2–4 моделей-кандидатов, прошедших hard constraints
2. Собрать тестсет из 3–5 реальных кейсов (70% типичных + 30% краевых)
3. Подготовить единый тест-промпт (одинаковые инструкции для всех моделей)
4. Переименовать ответы моделей в «Вариант A / B / C» и оценить вслепую по четырём метрикам
5. Выбрать основную модель и fallback (запасной вариант)

### Пояснение

Алгоритм из пяти шагов разобран в уроке 2.5 (блоки 2–3). Логика последовательная: сначала отбираем кандидатов (shortlist), затем готовим материал для теста (тестсет), формулируем единые условия (промпт), проводим слепую оценку (A/B/C без брендов), и на основании данных выбираем победителя + fallback.

### Почему именно такой порядок

- Нельзя собирать тестсет до shortlist — нужно знать, какие модели тестировать (вдруг у одной нет мультимодальности — тогда кейсы со сканами нерелевантны).
- Нельзя писать промпт до тестсета — промпт адаптируется под конкретные кейсы.
- Нельзя оценивать до промпта — без единых условий сравнение бессмысленно.
- Нельзя выбирать модель до оценки — решение принимается на основе данных, а не ожиданий.

## Вопрос 9

| Параметр | Значение |
|----------|---------|
| **Тип** | Кейс + множественный выбор (выбрать все ошибки) |
| **Цель Блума** | А-2.1 |
| **Урок** | 2.1, 2.4, 2.7 |
| **Сложность** | Сложный |
| **Критерий зачёта** | Минимум 2 из 3 правильных без лишних = зачёт |

### Кейс

Компания из сферы e-commerce (Россия, 50 сотрудников) решила использовать ИИ для генерации описаний товаров — 500 карточек в неделю. Выбрали frontier-модель от западного вендора: «Она первая во всех рейтингах, точно справится лучше всех».

Результат через три месяца:
- Стоимость генерации — в **пять раз** выше запланированного бюджета
- Юристы обнаружили, что данные товаров (включая непубличные характеристики) уходят на зарубежные серверы, и **заблокировали** проект
- **20% описаний** на русском — со стилистическими ошибками, которые приходится править вручную

### Вопрос

Какие ошибки выбора модели были допущены? Выберите **все** подходящие.

### Варианты ответов

- **A.** Переплата за frontier при простой массовой задаче — генерация карточек товаров не требует сложных рассуждений
- **B.** Не подключили RAG (retrieval-augmented generation) для улучшения качества описаний
- **C.** Проигнорировали hard constraint по приватности и локализации данных — данные ушли за рубеж
- **D.** Не подключили MCP-сервер для интеграции с каталогом товаров
- **E.** Не протестировали качество модели на русском языке перед внедрением
- **F.** Не обучили собственную модель на карточках товаров компании

### Правильные ответы

**A, C, E**

### Пояснение

Этот кейс — составной пример из реальных анти-паттернов, разобранных в уроке 2.1 (блок 3 — hard constraints; блок 4 — 6 ошибок выбора) и уроке 2.7 (типичные ошибки).

- **Ошибка A** — переплата. Генерация описаний товаров — массовая, относительно простая задача. Workhorse-модель или локальная платформа справились бы за долю стоимости. В уроке 2.1: «Как заказывать доставку пиццы на бронированном лимузине».
- **Ошибка C** — приватность. Данные ушли за рубеж, юристы заблокировали. Это hard constraint, который не был проверен до пилота. В уроке 2.1: «Прежде чем сравнивать качество — проверьте, что кандидат проходит ваши жёсткие ограничения».
- **Ошибка E** — русский не протестирован. 20% ошибок на русском — прямое следствие. В уроке 2.3: «Русский язык как отдельная проверка — переносить выводы с английского на русский без теста — риск».

### Почему другие не являются ошибками в данном кейсе

- **B** — RAG нужен, когда модели не хватает контекста из внешних источников. Для генерации описаний по карточкам товаров RAG не является обязательным решением и не устранил бы ни одну из трёх проблем.
- **D** — MCP — технология интеграции, которая не решает проблемы стоимости, приватности и русского языка. Это из другого модуля курса (модули 5–6).
- **F** — обучение собственной модели — переусложнение для задачи, которая решается правильным выбором класса модели и промптингом. Это не было ошибкой выбора.

## Вопрос 10

| Параметр | Значение |
|----------|---------|
| **Тип** | Кейс с вариантами ответа (1 из 4) |
| **Цель Блума** | А-2.1 |
| **Урок** | 2.1, 2.2, 2.7 |
| **Сложность** | Сложный |
| **Критерий зачёта** | 1 правильный ответ |

### Кейс

Банк (Россия, строгий комплаенс) хочет внедрить ИИ-помощника для операторов контакт-центра. Задача: генерация черновиков ответов клиентам (300 обращений в день).

**Hard constraints:**
- Персональные данные клиентов не могут покидать территорию РФ
- Нужен качественный русский язык (формальный стиль, юридически корректные формулировки)
- Нужен structured output: ответ должен содержать поля «приветствие», «суть ответа», «следующий шаг»
- Бюджет ограничен — решение должно быть экономически обоснованным при 300 обращениях/день

### Вопрос

Какой класс модели и контур использования **наиболее рационален** для этого кейса?

### Варианты ответов

- **A.** Frontier-модель глобального вендора через VPN — обеспечит максимальное качество ответов
- **B.** Privacy-first / локальная платформа в РФ-контуре — данные остаются в стране, модели оптимизированы под русский язык
- **C.** Open-weight модель без выделенной команды поддержки — бесплатная, значит, экономичная
- **D.** Специализированная ASR-модель для распознавания речи — она быстрее всех обрабатывает обращения

### Правильный ответ

**B**

### Пояснение

Анализ по алгоритму «задача → hard constraints → класс» (уроки 2.1–2.2):

- Hard constraint 1 (данные в РФ) → отсекает все варианты, где данные уходят за рубеж.
- Hard constraint 2 (русский язык) → нужна модель, оптимизированная или хорошо протестированная на русском.
- Hard constraint 3 (structured output) → нужна модель с надёжной поддержкой JSON.
- Hard constraint 4 (бюджет) → frontier нерационален при 300 обращениях/день.

Privacy-first / локальная платформа в РФ-контуре (GigaChat, YandexGPT) закрывает все четыре constraint: данные в РФ, оптимизация под русский, поддержка structured output, разумная стоимость.

### Почему другие неверны

- **A** — VPN не решает проблему комплаенса. Данные всё равно обрабатываются за рубежом. Для банка с ФЗ-152 это нарушение. Кроме того, ряд провайдеров не включает Россию в список поддерживаемых стран — обход ограничений не является бизнес-стратегией (урок 2.1).
- **C** — open-weight без команды поддержки — это риск эксплуатации. Self-hosted оправдан при наличии команды (урок 2.2). Без неё «бесплатная модель» может стать самой дорогой: простои, сбои, отсутствие обновлений.
- **D** — ASR-модель распознаёт речь, а задача — генерация текстовых ответов. Это другой класс компонентов, не решающий поставленную задачу.

## Что НЕ спрашивается в тесте

- Названия конкретных моделей (GPT-X, Claude Y) — volatile
- Конкретные цены и тарифы — volatile
- Списки стран доступности — volatile
- Техническая архитектура (трансформеры, attention) — не входит в модуль
- Промптинг — модуль 3
- RAG, агенты, MCP как отдельные темы — модули 5–6
- Академические бенчмарки — модуль учит, что бенчмарки ≠ замена тесту
