## Вопрос 1

**Тип:** один вариант из 3  
**Сложность:** лёгкий  
**Цель Блума:** З-6.1  
**Урок:** 6.1

### Текст вопроса

В компании настроена следующая система: когда клиент пишет в чат поддержки, система проверяет текст сообщения. Если в тексте есть слово «возврат» — автоматически создаётся тикет в CRM и отправляется уведомление менеджеру. Если клиент пишет «статус заказа» — система отправляет ссылку на трекинг. Если ни одно ключевое слово не распознано — обращение переадресуется оператору.

**К какой категории ИИ-решений относится эта система?**

- A) Ассистент (чат-бот)
- B) Workflow-автоматизация
- C) ИИ-агент

### Правильный ответ

**B) Workflow-автоматизация**

### Пояснение

Система работает по заранее определённым правилам «если → то»: конкретное ключевое слово → конкретное действие. Она не принимает решений самостоятельно, не вызывает инструменты динамически и не адаптирует поведение. Это классическая workflow-автоматизация — надёжная, предсказуемая, но неспособная справиться с нестандартными обращениями.

### Почему другие варианты неверны

- **A) Ассистент** — ассистент генерирует текстовый ответ и ждёт следующего запроса пользователя; он не выполняет действий (не создаёт тикеты, не отправляет уведомления).
- **C) ИИ-агент** — агент динамически выбирает следующий шаг на основе контекста; здесь же маршруты фиксированы заранее по ключевым словам.

**Цель Блума:** З-6.1  
**Урок:** 6.1

### Текст вопроса

Сопоставьте описания поведения системы (слева) с категорией ИИ-решения (справа). Каждая категория используется ровно два раза.

| # | Описание поведения | → | Категория |
|---|---|---|---|
| 1 | Система получает запрос пользователя, формирует текстовый ответ и ждёт следующего сообщения. Самостоятельно никаких действий во внешних системах не выполняет. | | Ассистент / Автоматизация / Агент |
| 2 | При возникновении ошибки система пробует другой подход, переформулирует запрос к инструменту или запрашивает дополнительные данные, прежде чем эскалировать на человека. | | Ассистент / Автоматизация / Агент |

| 6 | Система выполняет одну и ту же последовательность шагов для каждой входящей заявки: приём → проверка формата → регистрация → уведомление. Порядок не меняется. | | Ассистент / Автоматизация / Агент |

| # | Категория |
|---|-----------|
| 1 | Ассистент |
| 2 | Агент |
| 3 | Автоматизация |
| 4 | Агент |
| 5 | Ассистент |
| 6 | Автоматизация |

### Пояснение

Ключевые отличия: ассистент — только генерирует текст, не действует во внешних системах. Автоматизация — фиксированные правила, не адаптируется. Агент — динамически выбирает действия, работает с инструментами, корректирует план при ошибках. Описания 1 и 5 — ассистент (реактивный, без действий). Описания 3 и 6 — автоматизация (фиксированные цепочки). Описания 2 и 4 — агент (самокоррекция, динамический выбор инструментов).

- **#1 = Ассистент**: «формирует текстовый ответ и ждёт» — нет действий, нет инструментов (как в сценарии 6.1: «генерирует текст и ждёт следующего промпта»).

- **#3 = Автоматизация**: «если → то», «сбой при отклонении» — характеристика из таблицы сравнения.
- **#4 = Агент**: «сама решает, куда обратиться» + «выстраивает последовательность в зависимости от данных» — динамический выбор действий.
- **#5 = Ассистент**: не обращается во внешние системы, только текстовый ответ.
- **#6 = Автоматизация**: фиксированная последовательность, порядок не меняется.

## Вопрос 3

**Тип:** сопоставление (5 пар)  
**Сложность:** лёгкий  
**Цель Блума:** З-6.2  
**Урок:** 6.2  
**Принцип оценки:** ≥4 из 5 правильных = зачёт

### Текст вопроса

Сопоставьте агентный паттерн (слева) с бизнес-сценарием, который лучше всего ему соответствует (справа).

**Паттерны:**
1. Tool Use (вызов инструментов)
2. Planning (планирование)
3. Retrieval with Action (агентный RAG)
4. Reflection / Self-Review (самопроверка)
5. Supervised Multi-Step Workflow

**Бизнес-сценарии (перемешаны):**

- A) Агент получает заявку от поставщика, проверяет её по базе цен и условиям договора, оценивает достаточность информации, при необходимости запрашивает недостающие данные из другой системы и формирует отчёт о соответствии условиям закупки.
- B) Агент составляет проект коммерческого предложения, затем проверяет его по чек-листу: наличие обязательных разделов, корректность расчётов, соответствие шаблону компании. Находит несоответствие — исправляет и проверяет повторно.
- C) HR-агент получает запрос от сотрудника об остатке отпуска, самостоятельно обращается к кадровой системе, затем к платёжному модулю для расчёта компенсации и отправляет сотруднику ответ по email.
- D) Агент получает задачу «подготовить анализ рынка в новом регионе». Разбивает её на этапы: определить конкурентов, собрать данные по каждому, оценить позиционирование, сформировать итоговый отчёт. По ходу корректирует план, если по одному конкуренту данных недостаточно.
- E) Агент обрабатывает входящие счета от поставщиков: автоматически категоризирует, проверяет суммы по договору, но счета свыше определённого порога или с расхождениями передаёт бухгалтеру на одобрение.

### Правильные ответы

| Паттерн | Сценарий |
|---------|----------|
| 1. Tool Use | C |
| 2. Planning | D |
| 3. Retrieval with Action | A |
| 4. Reflection | B |
| 5. Supervised Multi-Step Workflow | E |

### Пояснение

Каждый паттерн определяется характерным поведением. Tool Use (C) — агент самостоятельно выбирает, к каким системам обратиться (кадровая, платёжный модуль, email) и вызывает их в рамках одного взаимодействия. Planning (D) — декомпозиция сложной задачи на шаги с адаптацией плана по ходу. Retrieval with Action (A) — агент не просто ищет информацию, а оценивает качество, запрашивает недостающее и действует на основе результатов. Reflection (B) — агент проверяет свой результат по критериям качества и итеративно улучшает. Supervised Workflow (E) — автоматическое выполнение рутинных шагов с передачей на человека в точках контроля.

### Почему другие варианты неверны (при ошибочном сопоставлении)

- Сценарий C — не Planning: агент не декомпозирует задачу, а последовательно вызывает конкретные инструменты.
- Сценарий D — не Tool Use: ключевое — составление и адаптация плана, а не просто вызов инструментов.
- Сценарий A — не Reflection: агент не проверяет собственный результат по чек-листу, а ищет и дополняет информацию.
- Сценарий B — не Supervised Workflow: нет передачи человеку, агент работает автономно в цикле самопроверки.
- Сценарий E — не Reflection: контрольная точка — одобрение человеком, а не самопроверка агентом.

## Вопрос 4

**Тип:** множественный выбор (выбрать 3 из 6)  
**Сложность:** средний  
**Цель Блума:** П-6.1  
**Урок:** 6.1, 6.4  
**Принцип оценки:** ≥2 из 3 верных + 0 ложных = зачёт. Если выбрано хотя бы 1 ложное — незачёт.

### Текст вопроса

**Ситуация:** Руководитель компании хочет внедрить ИИ-агента для обработки клиентских обращений. Коллега советует начать с supervised multi-step workflow — когда агент выполняет шаги автономно, но останавливается на контрольных точках для одобрения человеком — а не с полностью автономного агента.

**Какие из следующих аргументов КОРРЕКТНО объясняют, почему это хороший совет? Выберите 3.**

- A) Supervised workflow позволяет управлять рисками: человек контролирует критические шаги, где ошибка может привести к финансовым или репутационным потерям.
- B) Supervised workflow всегда дешевле любого другого агентного подхода, потому что требует меньше API-вызовов.
- C) Supervised workflow позволяет постепенно наращивать автономию агента — от read-only к рекомендациям, затем к действиям с подтверждением — на основе реальных данных о надёжности.
- D) Supervised workflow не требует guardrails, потому что человек и так контролирует каждый шаг.
- E) Supervised workflow обеспечивает human-in-the-loop на высокорисковых действиях, пока агент выполняет рутинные операции автоматически. Это баланс автоматизации и контроля.
- F) Supervised workflow — единственный агентный паттерн, одобренный для использования российским законодательством.

### Правильные ответы

**A, C, E**

### Пояснение

Supervised workflow рекомендуется как точка входа по трём причинам из уроков 6.1 и 6.4: (1) управление рисками через human-in-the-loop на критических шагах (A); (2) возможность progressive autonomy — постепенного наращивания самостоятельности агента на основе данных, а не интуиции (C); (3) баланс автоматизации рутины и контроля рисковых операций (E).

### Почему другие варианты неверны

- **B)** Неверно. Supervised workflow не обязательно дешевле — он может быть даже дороже за счёт времени человека на одобрение. Преимущество — в управлении рисками, а не в стоимости API-вызовов.
- **D)** Неверно. Supervised workflow требует guardrails так же, как любой другой подход. В уроке 6.4 показано: на каждой ступени автономии guardrails активны, меняется только их конфигурация.
- **F)** Неверно. Российское законодательство не «одобряет» конкретные агентные паттерны. 152-ФЗ регулирует обработку персональных данных, а не выбор архитектуры агента.

## Вопрос 5

**Тип:** один вариант из 4  
**Сложность:** средний  
**Цель Блума:** П-6.2  
**Урок:** 6.2

### Текст вопроса

**Ситуация:** Компания подключила ИИ к пяти внутренним системам: CRM, корпоративная почта, складской учёт, бухгалтерия и система тикетов. Каждая интеграция написана специально под модель, которую компания использует сейчас. Руководство рассматривает возможность смены LLM-провайдера на более подходящую модель.

**Какую проблему решает стандарт MCP (Model Context Protocol) в этой ситуации?**

- A) MCP ускоряет работу любой языковой модели за счёт оптимизации запросов к внешним системам.
- B) MCP объединяет CRM, почту, склад, бухгалтерию и тикеты в единый интерфейс, заменяя отдельные системы.
- C) MCP устраняет необходимость переписывать интеграции при смене модели: каждая сторона (модель и инструмент) реализует протокол один раз, и интеграции работают с любой совместимой моделью.
- D) MCP автоматически обучает новую модель на данных из подключённых систем компании, чтобы она работала не хуже предыдущей.

### Правильный ответ

**C)**

### Пояснение

MCP решает проблему N×M интеграций: без стандарта при 5 инструментах и смене модели нужно переписать все 5 интеграций. С MCP каждая модель и каждый инструмент реализуют протокол один раз — как USB-C для ИИ. Это снижает vendor lock-in: инвестиции в интеграцию сохраняются при смене модели.

### Почему другие варианты неверны

- **A)** MCP — стандарт подключения, а не инструмент оптимизации производительности моделей. Скорость модели определяется провайдером, не протоколом интеграции.
- **B)** MCP не заменяет корпоративные системы — он стандартизирует способ подключения ИИ к этим системам. CRM остаётся CRM, бухгалтерия — бухгалтерией.
- **D)** MCP не обучает модели и не адаптирует их под данные компании. Это протокол передачи данных между моделью и инструментами, а не механизм обучения.

## Вопрос 6

**Тип:** кейс + выбор ответа для каждой из 3 задач  
**Сложность:** средний  
**Цель Блума:** П-6.3  
**Урок:** 6.3  
**Принцип оценки:** ≥2 из 3 правильных = зачёт

### Текст вопроса

Компания (интернет-магазин, 80 сотрудников) рассматривает внедрение ИИ-агента для трёх задач. Для каждой определите **основной сигнал того, что агент избыточен**.

**Задача 1.** Отправка клиентам SMS-уведомлений о смене статуса заказа. Логика: «оплачен» → SMS «заказ принят», «собран» → SMS «заказ готов к отправке», «доставлен» → SMS «заказ доставлен». Шаблоны фиксированные.

**Задача 2.** Автоматическая генерация описаний товаров на основе характеристик из каталога. Проблема: в каталоге у 40% товаров отсутствуют ключевые характеристики (вес, материал, размеры), а у 15% — данные устарели или содержат ошибки.

**Задача 3.** Автоматическое одобрение возвратов без участия менеджера — агент принимает решение о возврате денег клиенту на основе фотографии товара и текста обращения, без верхнего порога суммы.

**Варианты сигналов:**
- (а) Процесс линеен и предсказуем — достаточно обычной автоматизации
- (б) Плохие данные — агент будет работать с мусором на входе
- (в) Высокая цена ошибки без human-in-the-loop
- (г) Нет владельца процесса
- (д) Agentwashing — хайп вместо реальной потребности

### Правильные ответы

| Задача | Сигнал |
|--------|--------|
| 1 | (а) Процесс линеен и предсказуем |
| 2 | (б) Плохие данные |
| 3 | (в) Высокая цена ошибки без human-in-the-loop |

### Пояснение

**Задача 1:** SMS-уведомления по фиксированным шаблонам при смене статуса — это классический линейный процесс «если → то». Агент здесь избыточен: обычная автоматизация (Zapier, n8n) справится быстрее, дешевле и надёжнее.

**Задача 2:** Когда 40% данных неполные и 15% ошибочные, любой агент будет порождать некачественный результат. Принцип из урока 6.3: «мусор на входе — мусор на выходе». Сначала нужно почистить каталог, потом думать об ИИ.

**Задача 3:** Автоматическое одобрение возвратов без порога суммы и без участия человека — это финансовое решение с потенциально высокой ценой ошибки. Агент может одобрить крупный мошеннический возврат. Необходим human-in-the-loop, как минимум для сумм выше определённого порога.

### Почему другие варианты неверны (для каждой задачи)

- Задача 1 не (б): данные (статусы заказов) структурированные и чистые — проблема не в данных, а в том, что процесс слишком простой для агента.
- Задача 2 не (а): генерация описаний — не линейный процесс; проблема конкретно в качестве данных.
- Задача 3 не (б): данные (фото, текст) могут быть нормальными; проблема в том, что финансовое решение принимается без человеческого контроля.

| 1 | Агент обработки заявок зациклился на одном шаге и за 6 часов совершил тысячи API-вызовов. Счёт от провайдера — значительно больше ожидаемого. |
| 2 | Агент ошибочно изменил данные в CRM — перезаписал контактную информацию у группы клиентов. Нужно восстановить данные до состояния перед ошибкой. |
| 3 | Агент получил доступ к финансовой системе и создал заказ поставщику на крупную сумму, которую никто не проверял. |
| 4 | Команда хочет запустить нового агента сразу в рабочую среду с реальными данными клиентов, не тестируя на ограниченном наборе данных. |

**Guardrails:**
- (а) Контроль затрат / бюджетные лимиты
- (б) Откат / rollback
- (в) Ограничения прав доступа / human approval
- (г) Песочница / sandbox

### Правильные ответы

| Ситуация | Guardrail |
|----------|-----------|
| 1 | (а) Контроль затрат |
| 2 | (б) Откат / rollback |
| 3 | (в) Ограничения прав доступа / human approval |
| 4 | (г) Песочница / sandbox |

### Пояснение

Каждый guardrail из урока 6.4 адресует конкретный тип инцидента. Контроль затрат (лимит API-вызовов, таймауты, circuit breaker) предотвращает финансовые потери от зацикливания. Откат позволяет вернуть систему в предыдущее состояние при ошибке с данными. Ограничения прав / human approval не дают агенту совершать высокорисковые финансовые действия без одобрения. Песочница — изолированная среда для тестирования до запуска с реальными данными.

### Почему другие варианты неверны (при ошибочном сопоставлении)

- Ситуация 1 — не (б): откат не решает проблему повторяющихся расходов, нужно ограничить количество вызовов и бюджет.
- Ситуация 2 — не (в): ограничение прав предотвратило бы ошибку заранее, но данные уже повреждены — нужен откат.
- Ситуация 3 — не (а): проблема не в стоимости API-вызовов, а в том, что агент совершил финансовое действие без одобрения.
- Ситуация 4 — не (б): откат пригодится после инцидента, а песочница предотвращает инцидент в первую очередь.

## Вопрос 8

**Тип:** кейс + один вариант из 4  
**Сложность:** сложный  
**Цель Блума:** П-6.3 + элемент Пр-6.1  
**Урок:** 6.3

### Текст вопроса

**Кейс:** Юридическая фирма (45 юристов) рассматривает внедрение ИИ для работы с претензиями клиентов. Каждая претензия уникальна: разные суммы (от 50 тыс. до 15 млн руб.), разные правовые основания, эмоциональный контекст клиента. Ошибка в ответе на претензию может привести к судебному разбирательству и репутационным потерям. Подготовка ответа требует анализа истории отношений с клиентом, проверки нормативных актов и учёта судебной практики.

**Используя матрицу McKinsey (риск × сложность суждения), определите рекомендуемый тип решения для этой задачи.**

- A) Полная автоматизация — агент обрабатывает претензии самостоятельно от начала до конца.
- B) ИИ-агент с лёгким надзором — агент работает автономно, руководитель просматривает результаты выборочно.
- C) Автоматизация с human approval gates — агент выполняет стандартные шаги автоматически, ключевые решения одобряет юрист.
- D) Человек с ИИ-копайлотом — юрист принимает решения, ИИ помогает с подготовкой информации и черновиков.

### Правильный ответ

**D) Человек с ИИ-копайлотом**

### Пояснение

По матрице McKinsey задача попадает в квадрант «высокий риск + высокое суждение». Риск высокий: ошибка ведёт к судебным разбирательствам и потерям до 15 млн руб. Суждение высокое: каждая претензия уникальна, требует юридической экспертизы, анализа контекста и эмпатии. Для такого квадранта рекомендуется «человек с ИИ-копайлотом» — юрист принимает решения, а ИИ помогает искать прецеденты, готовить черновики, проверять нормативную базу.

### Почему другие варианты неверны

- **A)** Полная автоматизация применима для задач с низким риском и низкой сложностью суждения (например, проверка статуса аккаунта). Юридические претензии — противоположный случай.
- **B)** Лёгкий надзор подходит для низкого риска при высокой сложности (например, персонализация рекомендаций). Здесь риск слишком высок для выборочного контроля.
- **C)** Автоматизация с human approval gates — для высокого риска при низкой сложности (например, одобрение платежей). Здесь сложность суждения высокая — человек должен быть в центре процесса, а не только на контрольных точках.

## Вопрос 9

**Тип:** порядок шагов (расставить 4 этапа)  
**Сложность:** средний  
**Цель Блума:** П-6.1  
**Урок:** 6.1, 6.4  
**Принцип оценки:** все 4 в правильном порядке = зачёт

### Текст вопроса

Компания внедряет ИИ-агента для обработки входящих заявок от клиентов. Ниже — четыре этапа наращивания автономии агента. **Расставьте их в рекомендуемом порядке, начиная с наименее автономного.**

Этапы (перемешаны):

- ( ) Агент самостоятельно обрабатывает заявки и выполняет действия в рамках установленных лимитов (например, отвечает на стандартные запросы, создаёт тикеты). Исключения — на менеджера.
- ( ) Агент анализирует заявку, предлагает вариант ответа и действий. Менеджер принимает или отклоняет предложение.
- ( ) Агент только читает входящие заявки, формирует сводку для менеджера. Никаких действий не выполняет.
- ( ) Агент выполняет действия по заявке (отправляет ответ, создаёт тикет), но каждое действие требует подтверждения менеджера перед исполнением.

### Правильный порядок

1. **Агент только читает входящие заявки, формирует сводку для менеджера.** (Read-only)
2. **Агент анализирует заявку, предлагает вариант ответа и действий. Менеджер принимает или отклоняет.** (Рекомендации)
3. **Агент выполняет действия, но каждое требует подтверждения менеджера.** (Действия с подтверждением)
4. **Агент самостоятельно обрабатывает заявки в рамках лимитов.** (Ограниченная автономия)

### Пояснение

Это принцип progressive autonomy из уроков 6.1 и 6.4: начинайте с максимального контроля, расширяйте автономию на основе данных, а не надежды. На каждом этапе набор guardrails разный, но они остаются активными всегда. Решение о переходе на следующий этап принимается на основе фактов (сколько ошибок, какая удовлетворённость клиентов), а не на основе интуиции.

### Почему именно этот порядок

- Read-only → рекомендации: сначала агент доказывает, что понимает задачу (качество сводок).
- Рекомендации → действия с подтверждением: менеджер убедился, что предложения адекватны.
- С подтверждением → ограниченная автономия: накоплены данные о надёжности на контролируемом этапе.
- Пропуск этапов (например, сразу к автономии) — ошибка Klarna: 66% обращений на агента с первого дня.

## Вопрос 10

**Тип:** кейс + множественный выбор (выбрать 3 из 6)  
**Сложность:** сложный  
**Цель Блума:** А-6.1  
**Урок:** 6.3, 6.4  
**Принцип оценки:** ≥2 из 3 верных + 0 ложных = зачёт. Если выбрано хотя бы 1 ложное — незачёт.

### Текст вопроса

**Кейс:** HR-отдел крупной компании (500+ сотрудников) внедрил ИИ-агента для первичного скрининга резюме и назначения собеседований. Агент работает так: получает резюме по email, анализирует текст, сопоставляет с требованиями вакансии, ранжирует кандидатов и автоматически отправляет приглашения на собеседование подходящим кандидатам. Агент также автоматически отправляет отказы тем, кто не прошёл скрининг.

Результаты за первый квартал: обработано 3 200 резюме, среднее время от получения до ответа сократилось с 5 дней до 4 часов, рекрутеры освободили 60% времени на другие задачи.

Детали внедрения: агент имеет полный доступ к корпоративной почте (чтение и отправка), базе вакансий и календарю рекрутеров. Ограничений на количество отправляемых писем нет. Все решения агента (приглашение/отказ) выполняются автоматически, без просмотра человеком.

**Какие 3 утверждения ВЕРНО описывают риски или недостающие guardrails в этом внедрении? Выберите 3.**

- A) У агента нет human approval на решениях об отказе — автоматический отказ сильному кандидату может привести к репутационным и юридическим потерям (дискриминация).
- B) Агенту необходимо добавить мультиагентную архитектуру: один агент анализирует резюме, второй принимает решение, третий отправляет письма — это повысит точность.
- C) Агент имеет неограниченный доступ к отправке писем — при ошибке или зацикливании может разослать тысячи некорректных сообщений от имени компании.
- D) Агенту нужно перейти на более мощную языковую модель, чтобы повысить качество анализа резюме — это решит проблему ошибочных отказов.
- E) Отсутствует ограничение прав доступа: агент может читать и отправлять любые письма через корпоративную почту, а не только связанные с рекрутингом.
- F) Агенту не хватает интеграции с социальными сетями кандидатов для более полного анализа — без этого скрининг неполный.

### Правильные ответы

**A, C, E**

### Пояснение

В описанном внедрении намеренно «пропущены» три критических проблемы:

**(A) Нет human approval на критических решениях.** Отказ кандидату — решение с юридическими и репутационными последствиями (потенциальные иски о дискриминации, потеря ценных кандидатов). По принципу из урока 6.4: если ошибка ведёт к юридическим или репутационным потерям — human approval обязателен.

**(C) Нет контроля затрат / лимитов на действия.** Без ограничений на количество отправляемых писем агент при зацикливании или ошибке может разослать массовые некорректные сообщения от имени компании — аналогично инциденту с 47 000 API-вызовов из урока 6.4.

**(E) Избыточные права доступа.** Агенту нужен доступ только к почте, связанной с рекрутингом, а не ко всей корпоративной почте. Принцип минимально необходимых разрешений из урока 6.4: наблюдение → рекомендация → исполнение, и только в рамках своей задачи.

### Почему другие варианты неверны

- **B)** Мультиагентная архитектура не решает описанные проблемы (отсутствие человеческого контроля и избыточные права). Мультиагентные системы не входят в scope модуля и не являются средством повышения надёжности в данном контексте.
- **D)** Смена модели не решает проблему отсутствия guardrails. Даже идеальная модель может ошибиться — и без human approval и лимитов ошибка приведёт к тем же последствиям.
- **F)** Анализ социальных сетей кандидатов создаёт дополнительные юридические и этические риски (конфиденциальность, дискриминация), а не решает описанные проблемы. Кроме того, это расширение функционала, а не guardrail.
